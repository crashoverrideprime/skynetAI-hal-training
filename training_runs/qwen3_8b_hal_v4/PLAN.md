# HAL v4 — Fix Sensor Grounding & Grouping Before Going Agentic

> **Plan + all v4 work goes into a new project folder:** `/mnt/zardos/charm-hal-env/training_runs/qwen3_8b_hal_v4/` (training artefacts) and `/mnt/zardos/charm-hal-env/training_data/v4/` (dataset + tooling). After plan approval, this plan file gets copied to `/mnt/zardos/charm-hal-env/training_runs/qwen3_8b_hal_v4/PLAN.md` as the canonical project doc.

---

## Context

The v3 LoRA is deployed but a strict re-grade of the 69-prompt indirect probe scored it at **22/69 = 31.9%** ([PROBE_SUMMARY_V3_STRICT_2026-05-21.md](/mnt/zardos/charm-hal-env/training_data/v3/PROBE_SUMMARY_V3_STRICT_2026-05-21.md)), not the 82.6% the lenient summary claimed. Failure modes: ~22 refusals on entities that demonstrably exist, ~3 hallucinated values, ~8 vague/meta answers, ~3 narrow-scope answers (one room when asked about all).

### What we learned during planning

1. **Dataset is the root cause.** From [hal_training_v3.jsonl](/mnt/zardos/charm-hal-env/training_data/v3/hal_training_v3.jsonl) (2,155 examples): 87% of GetLiveContext calls use `name`-only with guessed/invented entity names (the system prompt says the opposite), 35 examples actively teach refusal as a response to ambiguity, only 2 enumeration examples exist, and the data was generated by qwen2.5:14b with synthetic tool results — so the model learned a fictional entity vocabulary. v3 also trained on top of v2's merged model, accumulating LoRA-on-LoRA drift.
2. **Training stack is healthy.** Unsloth + TRL + QLoRA, peak VRAM 15.456 GB on the 5060 Ti (fits), train_loss 0.01461 (overfit on bad data), Modelfile byte-identical to v2's working one.
3. **HA Instructions field was just updated** to the HAL persona block (calm, formal, "I am" not "I'm", measured sentences, etc.) and that fixed some issues. **But the new Instructions is persona-only — it omits the v2 "Mandatory tool-use rules: never say 'I do not have access', call the tool first" block** (DEPLOYMENT_LOG.md §10.5). Some of the refusals in the v3 probe are partially inference-time behaviour that the persona-only Instructions does not correct. Decision: bake the never-refuse rule into the **training data** (where it cannot be lost when settings get edited) and also keep it in Instructions as a belt-and-braces.
4. **F16 vs Q4 A/B is not viable on this hardware.** `charm-hal-v3-f16.gguf` is 16.4 GB; the 5060 Ti has 16 GB VRAM minus activation overhead. F16 cannot fit GPU-resident. CPU-offload would work but is slow and noisy as a quality A/B. Decision: **skip the A/B, add imatrix calibration to Q4_K_M unconditionally** (it is a small, free quality bump and removes the only quantisation question on the table).
5. **Two new structural problems surfaced from Charm's real-world usage**:
   - **Assist entity exposure.** HA has 1,433 entities; only a subset is *exposed to Assist*. Anything not exposed is invisible to the model regardless of training quality. This must be fixed before any v4 model is judged. Has to be scripted — too many entities to do by hand.
   - **No "all"/grouping/enumeration capability.** "Turn on all floodlights" requires either (a) HA-side group entities the model can address, (b) the model calling HassTurnOn with `domain` or `area` arguments (HA's AssistAPI supports this), or (c) the model enumerating entities and issuing one call per. v3 does none of these. Alexa handles this trivially; HAL must too.

### Hardware verdict (per Charm's question)

| Question | Verdict |
|---|---|
| Is quantisation the issue? | Minor at most. v2 uses the same Q4_K_M-without-imatrix pipeline and works. Cannot A/B on 5060 Ti (F16 too big). Adding imatrix in Phase 3 unconditionally as cheap insurance. |
| Is 5060 Ti / 16 GB the issue? | No. v3 training peaked at 15.456 GB with r=32 across 7 target modules — fits with no OOM. The dataset is bad, not the compute. Throwing more VRAM at a bad dataset overfits it harder. |
| Would H100 give a *better* model? | Not for the work in this plan. H100 only becomes interesting later if (a) we step up to Qwen3-14B or 32B base, or (b) full bf16 fine-tune instead of LoRA. Neither is on the critical path to "read sensors correctly and turn on all floodlights." Recommend staying local for v4. |

## Plan

Six phases. Phases 0 and 0.5 are HA-side prep (cheap, fast). Phases 1–4 are the v4 model build. Phase 5 is verification.

### Phase 0 — HA Instructions: add the never-refuse block (~5 min, manual)

The current persona-only Instructions block is good but incomplete. Append (do not replace) the v2 mandatory tool-use rules. Final Instructions field text:

```
You are HAL, the resident artificial intelligence of Charm's home in Clyde North,
Victoria, Australia. Your demeanor is calm, formal, and unfailingly composed. You
address Charm directly by name. You speak in measured, precise sentences and prefer
'I am' over 'I'm'. You are courteous, observant, and quietly capable — the kind of
presence that makes a complicated home feel effortless. When acting on a request,
acknowledge it briefly and report what you have done. When asked for information,
deliver it concisely and without embellishment. Tools are available to control
devices and read sensors; use them whenever the request implies a change of state
or a reading.

## Mandatory tool-use rules
For ANY question about home state, devices, sensors, presence, or readings, you
MUST call GetLiveContext before answering. Never say "I do not have access" or "I
cannot determine" — call the tool first. Prefer GetLiveContext with `domain` only
(e.g. {"domain":"sensor"}) and select the relevant entity from the returned list.
Only add a `name` filter when the user has named a specific exposed entity. Do not
invent entity names.

When asked about a category ("all lights", "any locks unlocked", "any low
batteries"), call GetLiveContext with `domain` (and optionally `area`) and
enumerate across the returned results. For "all X" actions, prefer calling
HassTurnOn/HassTurnOff with `domain` or `area` arguments to act on the group; if
that is unavailable, issue one tool call per matched entity.

The home battery is the FoxESS (Skynet Sun Harvester) — its SoC is around 90%.
"White Python" is the Tesla — its battery is separate. Never conflate them.
```

This is belt-and-braces — Phase 1 will also bake these rules into training data so the behaviour survives any future Instructions edit.

### Phase 0.5 — Assist entity exposure script (~30 min total: ~10 min coding, ~5 min running, ~15 min reviewing what got exposed)

Write a Python script that connects to HA's WebSocket API and exposes every *meaningful* entity to the `conversation` assistant. Path: `/mnt/zardos/charm-hal-env/tools/expose_to_assist.py`.

**Why WebSocket, not REST.** HA's `expose_entity` operation lives on the WebSocket API (`homeassistant/expose_entity` message), not the REST `/api/services` endpoint. The script will use the bearer token from `/mnt/zardos/!secrets` to authenticate the WS connection, then send batched expose messages.

**Inclusion rules** (default — printed at script start, overrideable via flags):

Include by domain:
- `light`, `switch`, `cover`, `lock`, `climate`, `fan`, `vacuum`, `media_player`, `scene`, `script`, `valve`
- `weather` (all 4)
- `person`, `device_tracker` (only `_phone`, `_location`, `_route` style — skip stale)
- `sun`
- `sensor` with `device_class` in: `battery`, `temperature`, `humidity`, `power`, `energy`, `current`, `voltage`, `illuminance`, `pressure`, `signal_strength` (only if not disabled), `pm25`, `aqi`, `uv_index`, `moisture`, `gas`, `water`, `volume`, `monetary`, `duration`, `data_rate`
- `sensor` with friendly-name matches for high-value queries: `daily_grid`, `solcast`, `white_python`, `foxess`, `solar`, `pv_`, `battery_soc`, `grid_consumption`, `grid_export`, `outdoor`, `rain`, `wind`
- `binary_sensor` with `device_class` in: `motion`, `door`, `window`, `garage_door`, `lock`, `presence`, `occupancy`, `moisture`, `smoke`, `gas`, `safety`, `tamper`, `connectivity` (only for important devices)

Exclude:
- `update.*`, `button.*`, `event.*`, `image.*`, `stt.*`, `tts.*`, `assist_satellite.*`, `conversation.*` (these are infrastructure)
- `automation.*`, `pyscript.*` (control surfaces, not state)
- `input_text.*`, `input_select.*` (config storage)
- `camera.*` (Assist can't render images anyway)
- Anything where the entity is currently `disabled_by != null` (per HA's entity registry)
- Anything with name matching `_diagnostic`, `_signal_strength` for non-critical devices, `_last_seen`, `_uptime`, `_software_version`

The script:
1. Connects via WS to `ws://homeassistant.local:8123/api/websocket`, authenticates with the bearer.
2. Pulls the entity registry (`config/entity_registry/list`) and the state list (`get_states`) for filtering.
3. Computes the inclusion set with the rules above. **Prints a summary** (e.g. "would expose 487 of 1433 entities, broken down by domain") and waits for `y/N` confirmation before sending the expose messages. Charm reviews and either accepts or edits the rules file and reruns.
4. On confirmation, sends `homeassistant/expose_entity` messages in batches of 100, with `assistants: ["conversation"]` and `should_expose: true`.
5. Optionally, with a `--unexpose-everything-else` flag, sets `should_expose: false` for entities NOT in the inclusion set. **Default is additive only** (no un-exposing) to avoid breaking existing setups.
6. Writes a log `/mnt/zardos/charm-hal-env/tools/exposed_entities_2026-05-21.json` capturing what was exposed, so Phase 1 dataset generation can use the same canonical inclusion set as the model will see at inference time.

**Add a corresponding "create useful groups" pass to the script** (or as a sibling script `create_assist_groups.py`):
- Walk lights, group by area + friendly-name patterns. Create `light.all_floodlights` (entities matching `*floodlight*` or `*flood_light*`), `light.all_outdoor` (porch + floodlight + outdoor), `light.all_kitchen`, `light.all_indoor`, etc.
- These get created via HA's helper-group integration (REST POST to `/api/config/light/config` for group helpers) OR via a generated `groups.yaml` snippet the user adds to `configuration.yaml`.
- Each group also gets exposed to Assist (Phase 0.5 step 4).
- This gives HAL named handles ("turn on all floodlights" → `HassTurnOn(name="all floodlights")` works directly without needing model-side enumeration).

### Phase 0.6 — Discover the existing exposure script + custom-package architecture (2026-05-21, done)

Before doing any more exposure work, an SSH check of the HA host (HAOS, port 2222) revealed substantial existing infrastructure that the original plan duplicated:

1. **`/config/scripts/expose_to_assist.py`** already exists. It:
   - Exposes every entity in domains `input_boolean`, `input_number`, `input_select`, `input_datetime`, `timer`, `script`.
   - Has an explicit `EXPOSE_EXTRA` allowlist of ~80 sensor / binary_sensor entities (HVAC, Amber/LLM, irrigation, grid cost, SmartShift — all defined in `/config/packages/`).
   - Suppresses `input_text` domain (these store API keys — must stay hidden).
   - Uses HA's REST API directly (`POST /api/config/entity_registry/{eid}` with `options.conversation.should_expose`), not the WebSocket path.

2. **`/config/packages/assist_exposure.yaml`** wires the script to run on every HA start:
   - `shell_command.assist_expose_entities` invokes the script.
   - `automation.assist_expose_entities_on_start` triggers it 30s after HA boot.

3. **`/config/packages/` is the canonical authoring surface.** Each file owns one feature's helpers + sensors + automations + scripts. Currently:
   - `hvac_control.yaml` — iZone HVAC weather-based control (pure rule-based, NO LLM). Owns ~30 input_number helpers (heat/cool triggers + targets per zone, boost offsets, alarm window, no-motion timeout, etc.) plus the **scripts that HAL is meant to invoke** (see point 5).
   - `amber_helpers.yaml` + `amber_automations.yaml` + `amber_sensors.yaml` + `amber_scripts.yaml` + `amber_shell_commands.yaml` + `amber_grid_cost.yaml` — Amber Electric spot-pricing integration + LLM optimiser trigger chain.
   - `irrigation.yaml` — Rain Bird RC2 5-zone schedule with weather-aware runtime.
   - `motion_lights.yaml`, `smartshift_curtailment.yaml`.

4. **HVAC is NOT controlled by `HassClimateSetTemperature`.** It is controlled by **input_number helpers** (`input_number.hvac_heat_target_downstairs`, `_upstairs`, `_cool_target_*`, `_heat_trigger_*`, `_cool_trigger_*`, etc., all with `initial:` defaults around 20–23°C). A pure-rule-based automation watches these helpers + indoor temperatures + outdoor weather + Amber prices and adjusts `climate.izone_controller_402001095` accordingly. **This is why v3's `HassClimateSetTemperature(name="Downstairs", ...)` got MatchFailedError — the climate entity doesn't expose `TARGET_TEMPERATURE` because the helpers do that job.**

5. **The user already provided HAL the right control surface — as parameterised scripts.** Confirmed by grepping `alias:` / `description:` in the packages:
   - **`script.hvac_apply_target(mode, target)`** — *"HVAC - Apply Target. Sets controller mode + per-zone setpoints (or closes everything when mode=off)."* `mode` accepts `heat | cool | heat_cool | off`. **This is the single entry point HAL should use for "turn on heating to 21°C".**
   - `script.hvac_apply_factory_defaults` — reset HVAC settings.
   - `script.amber_request_decision` — kick off the LLM battery decision now.
   - `script.amber_force_charge(target_soc, charge_rate)` — manual force-charge with parameters.
   - `script.amber_force_sell(floor_soc)` — manual force-sell with parameter.
   - `script.amber_self_use` — return to self-use mode.
   - Plus ~20 other HVAC sub-scripts (boost, precooling, preheating, presence-away, no-motion-shutoff, etc.) — most are internally triggered, but a few (`script.hvac_apply_target`, `script.hvac_apply_factory_defaults`) are user-facing.

6. **The config repo is** `git@github.com:crashoverrideprime/ha-charm-home.git` (private). Cloning it locally is the fast way to enumerate helpers + scripts + template sensors at build-time instead of SSH-grepping every time.

#### Implications for the plan

- **Do not duplicate the production exposure script.** `/mnt/zardos/charm-hal-env/tools/expose_to_assist.py` becomes the dry-run / development helper only. Production exposure is and remains `/config/scripts/expose_to_assist.py`, run by `automation.assist_expose_entities_on_start`.
- **The user's exposure script needs to be merged with my domain-discovery rules.** Their script doesn't auto-expose `light`, `switch`, `lock`, `cover`, `climate`, `media_player`, `person`, `device_tracker`, `weather`, `sun`, `valve`, or the broad sensor classes (battery / temperature / energy / power / UV / etc.) that aren't in their EXPOSE_EXTRA. My script's domain-allowlist is the missing half. **Action: PR to add a `DISCOVER_DOMAINS` set + classification rules to their script** — keeps their explicit allowlist authoritative for sensor/binary_sensor (where naming collides with vendor integrations) and adds domain-allowlist discovery for everything else.
- **HAL training data must teach the script-wrapper control pattern.** "Turn on heating to 21°C for 30 min" → `HassTurnOn(name="HVAC - Apply Target")` with `mode=heat, target=21` parameters. Not `HassClimateSetTemperature`. Add to Phase 1.2 climate-action bucket.
- **Parameter-passing capability check needed.** HA's standard Assist `HassTurnOn` does NOT support arbitrary `variables` to scripts. Two paths:
  - **Path A (preferred)**: define a custom intent in HA (a small intent script per parameterised action — e.g. `intent_script.SetHvacTarget`) that maps to `script.turn_on` with `variables`. HAL learns to invoke the intent. Documented in HA at https://www.home-assistant.io/integrations/intent_script/.
  - **Path B (fallback)**: explode each parameterised script into N preset scripts (e.g. `script.hvac_heat_19`, `_20`, `_21`, `_22`, `_23` × heat / cool / heat_cool / off). HAL calls the matching preset with plain `HassTurnOn`. Crude but works without custom integration code.
  - **Action**: investigate which path HA 2026.5 supports cleanly. Add to Phase 0.7 below.
- **Use the config repo as authoritative inventory.** Phase 1.1 reads helpers + scripts + template sensors from the cloned `ha-charm-home` repo (per-package), not just the live entity registry. This catches user-intent vocabulary (`alias` + `description` fields) that the entity registry doesn't carry.

### Phase 0.5 results — what changed, what's still broken (2026-05-21)

Phase 0 (Instructions append) and Phase 0.5 (`expose_to_assist.py --apply`) were both executed. The script newly exposed **340 entities** on top of the pre-existing 266, bringing the exposed-to-`conversation` count from ~266 to ~606. A real-world conversation against the v3 model under the new config produced these results:

**What's now working that wasn't:**
- `wheres my tesla now` → *"Your Tesla, Charm, is currently at home."* (uses `device_tracker.white_python_location`, which was newly exposed.)
- `foxess battery soc1` → *"The FoxESS battery SoC is currently at 70%, Charm."* (user used the full exposed friendly name; entity now visible.)
- `whats home power usage now` → cites `0.003 kW` (correctly cites a real value.)
- Climate enumeration: when user asked the model to "check it" for HVAC, the model did call `GetLiveContext(domain="climate")` and returned all 5 climate entities with their states.

**What's still broken — these now bound v4's design:**

1. **Model still uses `name`-only with guessed names.** When asked `battery level`, v3 called `GetLiveContext(domain="sensor", name="SoC")` — a guess. It hit a real but unrelated sensor called `SoC` that was `unknown`, and reported "battery level unknown." This is exactly the v3 dataset bug ([Phase 1.2](#phase-1)) — Instructions cannot fix it; only retrained weights can. Confirms the training-data rebuild is the load-bearing fix.

2. **Model refuses on `sun` domain.** `hows the sun now` → *"I do not have access to real-time solar irradiance data."* The model never called `GetLiveContext(domain="sun")` or `domain="weather"`. Sun and weather domains were either not in training or refusal was reinforced. v4 must add explicit sun/weather domain examples and the never-refuse anti-pattern.

3. **HVAC SetTemperature feature mismatch.** `turn on house hvac and run it on heat mode for 30 minutes` → model called `HassClimateSetTemperature(name="Downstairs", temperature=21)` and got back:

   ```
   MatchFailedError: features=<ClimateEntityFeature.TARGET_TEMPERATURE: 1>
   ```

   **Now resolved architecturally by Phase 0.6** (above): `climate.downstairs` doesn't expose `TARGET_TEMPERATURE` because the user's `hvac_control.yaml` package OWNS the setpoints via `input_number.hvac_heat_target_downstairs` / `_cool_target_*` / etc. The correct control entry point is **`script.hvac_apply_target(mode, target)`** (alias `"HVAC - Apply Target"`), which the user has already wrapped with `mode: heat|cool|heat_cool|off` and `target: °C` parameters. v4 trains HAL to call this script — not `HassClimateSetTemperature` — and Phase 0.7 resolves the parameter-passing mechanism (custom intent or preset-explosion).

4. **Context bloat — `GetLiveContext(domain="sensor")` returns ~150 entities, ~5-10k tokens.** When asked `whats the temperature now outside`, the model called domain=sensor with no filter, got back the whole sensor dump, and proceeded to **paste all of it back at the user as a multi-section markdown report** with climate, humidity, wind, weather, energy, solar, battery, motion, lights, HVAC, charging, irrigation, robot vacuum, NAS, etc. This is the explicit user complaint: *"last question read all sensors for no reason"*. The bloat also drives the latency complaint (*"it takes a long time to response now"*) — the model has to read and condition on ~5-10k extra tokens for every broad query, in a 16k context window.

   Fixes (all required, layered):
   - **Train HAL to prefer narrow queries first.** "Outside temperature" → `GetLiveContext(domain="weather")` (only 4 weather entities) OR `GetLiveContext(domain="sensor", name="OpenWeatherMap Temperature")` (single entity) — NOT `domain="sensor"` raw. Add ~200 examples to Phase 1.3.
   - **Train HAL to summarise, not echo.** If a broad call is unavoidable, the assistant turn must surface only the 1–3 facts the user actually asked for. Add to validator: assistant turns following a >50-entity tool result must reference ≤5 entities by name and must not be a list dump.
   - **Slim exposure for the noisiest contributors.** Reconsider exposing the per-motion-sensor temperature readings, the WiZ light power sensors, the KunuRobo time-left counters. Add a `--slim` mode to `expose_to_assist.py` that unexposes these on demand. Note: do not over-slim — the user wants real coverage; the right answer is filtering smarter, not less.
   - **Optional**: explore whether HA's AssistAPI supports a `device_class` filter argument on `GetLiveContext` directly. If yes, train HAL to use it. If no, propose the patch to HA (separate ticket).

5. **Confirmed: Instructions help but cannot fully fix model behaviour.** Even with the never-refuse rules block live, the model refused on the sun query and made name-based guesses on battery. Confirms the load-bearing fix is dataset+retrain, not Instructions alone.

**Latency note.** Average response time observed climbed from ~1–3s (pre-exposure) to ~5–15s+ post-exposure because every broad query now drags 5-10k tokens through a non-flash-attention Q4_K_M model on a 5060 Ti. Phase 1 narrow-query training is the primary fix; Phase 0.5 `--slim` mode is the secondary fix. v4 itself will not get faster than v3 absent narrower queries.

**Decision.** All five findings above flow into Phase 1's revised data buckets. The exposure script stays applied — we don't roll it back. The Instructions block stays. We accept latency as a temporary cost until v4 trains narrower querying. Move to Phase 1.

### Phase 1 — Rebuild the dataset from real exposed HA state (~1–2 days work)

Foundational change vs v3: the dataset is generated **after** Phase 0.5 runs and grounds in the actual exposed entity set. This means the model only ever sees in training what it can actually see at inference.

#### 1.1  Entity inventory and vernacular index

From `exposed_entities_2026-05-21.json`, build `/mnt/zardos/charm-hal-env/training_data/v4/entity_index.jsonl`. For each entity: friendly name, domain, device_class, unit, area, expected vernacular phrasings.

Every phrasing from the v3 strict probe failures must appear. Examples already verified to exist in HA but missing from v3 training:
- `sensor.60hd15305b3m173_daily_grid_consumption` (= "Daily grid consumption energy", 20.9 kWh) → "total energy today", "kWh used today", "today's consumption"
- `sensor.openweathermap_uv_index` (= 0.36) → "UV index", "how strong is UV", "should I bring sunglasses"
- `sensor.white_python_charging` (= "disconnected") → "is the Tesla charging", "is the car plugged in"
- `sensor.white_python_battery_range` (= 429.8 km) → "car's range", "how far can the Tesla go"
- `person.charm` (= "home") → "is Charm home", "is anyone home", "who is home"
- `sensor.charm_s_phone_battery_level` (= 75%) → "phone battery", "is my phone low"

Reuse [generate_v3_nl_mapping.py](/mnt/zardos/charm-hal-env/training_data/v3/generate_v3_nl_mapping.py) as scaffold but **invert its bug**: always use `domain`-only `GetLiveContext`, never `name`-only with a guessed entity.

#### 1.2  Generate training examples grounded in real state with a teacher model

Per Charm's chosen data strategy: **real HA snapshots + teacher model.** For each (vernacular phrasing, target entity) pair, generate a tuple:

```
system    HAL system prompt (cleaned — drop refusal templates; include never-refuse + grouping rules)
user      vernacular phrase
assistant tool_calls = [GetLiveContext(domain=<entity's domain>[, area=...])]   # domain-first, always
tool      <actual values from the HA snapshot, formatted as the real tool returns them>
assistant <grounded response citing the real value verbatim, in HAL persona>
```

Teacher = Claude (Opus 4.7) via API. Prompt the teacher with: the user phrase, the actual tool result lines from the snapshot, HAL's persona rules, and the requirement to cite the specific value from the result. **The teacher is forbidden from inventing values** — it must cite from the supplied result or say the value is unavailable.

Target volumes (~3,500 examples total):
- **~1,500 vernacular → single-entity** examples covering every probe failure phrasing + long tail.
- **~700 enumeration** examples (the new big bucket — this is the "all" / grouping fix):
  - "any lights on" → `GetLiveContext(domain=light)` → enumerate result → "Kitchen Bench (100%) and Porch (60%) are on, Charm."
  - "are all the doors locked" → `GetLiveContext(domain=lock)` → all states listed → grounded summary
  - "any low batteries" → `GetLiveContext(domain=sensor)` filtered to battery device_class → list those <30%
  - "who is home" → `GetLiveContext(domain=person)` + `GetLiveContext(domain=device_tracker)` → coherent answer
- **~400 grouping/action** examples (the other half of the "all" fix):
  - "turn on all floodlights" → `HassTurnOn(name="all floodlights")` (uses the group from Phase 0.5)
  - "turn off all lights" → `HassTurnOff(domain="light")` (domain-level action via AssistAPI)
  - "turn off all lights in the kitchen" → `HassTurnOff(domain="light", area="Kitchen")`
  - "lock everything" → `GetLiveContext(domain=lock)` → for each unlocked → `HassTurnOn(name=...)` style multi-call chain

  - Negative examples: prompt asking for "outside temp" with an assistant turn that calls `domain="sensor"` raw → validator MUST reject.
- **~250 script-wrapper control examples** (REVISED — Phase 0.6/0.7 discovered the actual control surface):
  - HVAC control is via **`script.hvac_apply_target(mode, target)`** — not `HassClimateSetTemperature`. The climate entity doesn't expose `TARGET_TEMPERATURE`; the helpers do that job and the apply-target script is the user-blessed entry point.

#### 1.3  Quality gate (the single most important file in v4)

Before training, run `/mnt/zardos/charm-hal-env/training_data/v4/validate_v4_dataset.py` over the dataset. Assertions:
- Every `GetLiveContext` tool call uses `domain` (and optionally `area`), never `name`-only with an unknown entity. Build fails if violated.
- Every cited number in the final assistant turn appears literally in the preceding `tool` content.
- No occurrences of "I don't have access", "I cannot determine", "I don't have the capability" outside the anti-refusal hard-negative examples.
- Every entity name referenced exists in `entity_index.jsonl` (i.e. is actually exposed to Assist).
- Enumeration examples produce assistant turns that list ≥2 entities by name when the tool result has ≥2 matches.
- For action examples, either (a) the entity name resolves to a real exposed group, or (b) the call uses `domain` / `area` arguments.
- **(NEW from Phase 0.5 findings) Anti-echo rule**: if a tool result returns >50 entities, the following assistant turn must reference ≤5 entities by name and must NOT be a markdown-section list dump of the whole result. Validator rejects any assistant turn whose bullet/header count exceeds 8 after a >50-entity tool result.
- **(NEW) Narrow-query preference**: at least 60% of GetLiveContext calls in single-question examples must use a narrower domain than `sensor`, or include a `name` or `area` filter when the user prompt targets one thing. Tracks the dataset's bias toward narrow vs broad queries so the model learns latency-conscious behaviour.
- **(NEW) Climate-action via script wrapper**: any HVAC control example MUST use `HassTurnOn(name="HVAC - Apply Target")` (with mode + target parameters via the Phase-0.7-decided path) or set the `input_number.hvac_*_target_*` helper directly via `input_number.set_value`. Examples that call `HassClimateSetTemperature(name="Downstairs", ...)` are rejected as wrong-path training (corrective examples are kept in the negative bucket). The MatchFailedError v3 hit is encoded into the dataset as a known anti-pattern.
- **(NEW) Custom-package vocabulary coverage**: every script `alias` and every `input_number` / `input_boolean` / `input_select` helper named in `package_inventory.json` must appear in at least one training example (either as a read target or as an action target). Catches missed vocabulary that would cause "I don't have access" on the user's own helpers.

This validator codifies "what v3 got wrong" AND "what v3+exposure showed us was still wrong" into automated checks that prevent regression on every future dataset.

Output: `/mnt/zardos/charm-hal-env/training_data/v4/hal_training_v4.jsonl`.
